# NextSession Mass-Deployment Runbook

Rolling out the **NextSession** remote-access agent to ~1000 endpoints
(Windows + Linux + macOS) via an RMM (NinjaOne, Datto, ConnectWise Automate,
Action1, Intune, Jamf, etc.).

Servers, relay, and key are **baked into the signed `custom.txt`** build of the
binary. The provisioning scripts therefore **never** configure servers. Do not
pass `--config`/`--custom-server` anywhere — it would override the baked-in
servers.

Scripts referenced by this runbook:

- `deploy/provision-windows.ps1`
- `deploy/provision-linux.sh`
- `deploy/provision-macos.sh`
- `deploy/macos-mdm-pppc.md` (the load-bearing macOS prerequisite)

---

## 1. Access model: unique-per-device password + shared address book

### What we are building

Every endpoint gets a **unique, strong, permanent unattended password**
generated locally at provision time. The device ID + hostname alias + tags + that
saved password are pushed into **one shared address book** ("RMM Fleet")
that every technician sees. A tech opens the address book, double-clicks the
host, and connects one-click — no password typing, no per-device password
sheet to maintain.

### Why unique-per-device, not one shared password

| | One shared password | Unique-per-device (this design) |
|---|---|---|
| Blast radius if leaked | **Entire fleet** — one secret unlocks 1000 boxes | One device |
| Rotation | Must touch all 1000 at once | Per-device, independent |
| Offboarding a tech | Useless unless you rotate the whole fleet | Revoke their address-book share; passwords stay sealed |
| Tech convenience | One-click | One-click (the AB carries the per-device secret) |

The shared address book is what makes unique-per-device passwords usable at
scale: the per-device secret lives **in the AB entry**, so techs never see or
type it. They get the convenience of a shared password with the blast-radius of
per-device secrets.

### What the API actually supports (per findings)

Confirmed against the apiserver fork (`model/addressBook.go`,
`service/addressBook.go`, `http/controller/admin/*`, `http/router/admin.go`):

- A **shared collection** is an `AddressBookCollection` owned by a user; it is
  shared to others via an `AddressBookCollectionRule` (`Type=2` group share,
  `Rule>=1` read / `Rule=2` read-write).
- An admin service-account token (`api-token` header; the bound user **must be an
  admin**) can create entries unattended via
  `POST /api/admin/address_book/create` with
  `{ id, user_id, collection_id, alias, hostname, username, platform, tags[],
  password, hash:"" }`.
- For a **shared** AB the saved connection secret lives in the **`password`**
  field (the personal-AB `hash` field stays empty).
- **The server stores `password` verbatim — no server-side crypto.** Whatever
  bytes you POST are exactly what the client receives back
  (`service/addressBook.go` has no encryption; `PeerUpdate` writes the field
  raw). Any encryption of the saved secret is the client's responsibility.

So the scripted path is sound: one-time create the shared collection + group
rule, then per-device `address_book/create`.

### The one caveat — verify before mass rollout

Because the server stores `password` verbatim, whether a **plaintext** value in
the shared-AB `password` field yields working one-click connect depends on the
**client** version (some clients expect the value verbatim, some wrapped). This
lives in the client, not the server, so it **cannot be confirmed from the repo.**

> **Mandatory gate:** push **one** entry, open the client as a tech, confirm
> one-click connect succeeds, **before** scripting 1000. See Pilot (§4).

### Fallback if the live test fails

1. **Push entries without `password`** — techs still get ID/alias/tags/one-click
   discovery; the secret lives only in the RMM/CMDB and the tech pastes it. (Set
   the scripts to omit `password` / use `--no-register` and rely on the
   CSV/stdout record.)
2. **`batchCreateFromPeers`** — populate from peers that already self-registered
   via heartbeat, then patch tags/alias.
3. **Client-importable `.json` address book** generated by the RMM and imported
   client-side (bypasses the server for the password concern entirely).

## 2. Prerequisites

### 2.1 Service account in the API server (least privilege)

The admin AB endpoints require an **admin** user's token in the `api-token`
header (`middleware/admin_privilege.go` → `IsAdmin`). There is no
non-admin write path to a shared collection owned by another user, so the
registration account **must be admin**. Minimize its exposure:

1. Create a dedicated user, e.g. `svc-rmm-provision`, marked admin and enabled.
2. Mint a long-lived API token for it (`UserToken`). This token is the
   `NEXTSESSION_API_TOKEN` / `NS_API_TOKEN` / `NS_ADMIN_API_TOKEN` the scripts
   read **from the environment** (never hardcoded).
3. One-time, as that account:
   - Create the shared collection:
     `POST /api/admin/address_book_collection/create`
     `{ "name":"RMM Fleet", "user_id":<svc_user_id> }` → note the collection id.
   - Share it to the techs' group:
     `POST /api/admin/address_book_collection_rule/create`
     `{ "user_id":<svc_user_id>, "collection_id":<cid>, "type":2,
     "to_id":<tech_group_id>, "rule":2 }`.
4. Record `<svc_user_id>` and `<cid>` — they are required script inputs
   (`-ServiceUserId`/`-CollectionId`, `NS_AB_USER_ID`/`NS_AB_COLLECTION_ID`).

> Least-privilege note: this account is admin-broad because the API offers no
> finer grant for cross-user AB writes. Treat its token as a fleet-wide secret
> (see §5). Scope its blast radius operationally: store it only in the RMM secret
> store, rotate it on a schedule, and disable it the moment provisioning is done
> if you are not doing continuous re-registration.

### 2.2 Installer artifacts per OS

All artifacts must be the **signed `custom.txt` (NextSession-branded)** builds.

| OS | Artifact | Notes |
|---|---|---|
| Windows | `nextsession-host.exe` (renamed host exe) | Staged on a UNC share or HTTPS URL the RMM can reach. |
| Linux | `nextsession.deb` (Debian/Ubuntu) | `apt install ./file.deb` resolves deps. RPM path: adapt `install_deb`. |
| macOS | Signed + **notarized** `NextSession.pkg` | Postinstall must drop the LaunchDaemon + LaunchAgent. **Plus** the PPPC profile (§3). |

### 2.3 RMM job setup

- Run **elevated / as SYSTEM** (Windows) or **root** (Linux/macOS). The agent's
  privileged CLI (`--password`, `--option`, `--get-id`) requires
  `is_installed() && is_root()` and a **running service** — the RMM agent's
  SYSTEM/root context satisfies this.
- Inject the admin token as an **environment variable / secret field**, never as
  a literal in the policy body:
  - Windows: `$env:NEXTSESSION_API_TOKEN = "<token>"`
  - Linux: `NS_API_TOKEN=<token>` (plus `NS_AB_USER_ID`, `NS_AB_COLLECTION_ID`)
  - macOS: `NS_ADMIN_API_TOKEN=<token>` (plus `NS_API_BASE`, `NS_AB_USER_ID`,
    `NS_AB_COLLECTION_ID`)
- Configure the RMM to **collect the per-host record** (stdout line
  `NEXTSESSION_RECORD {...}` / the CSV) so passwords are reconcilable even if the
  API write is skipped or fails.
- The scripts are **idempotent** — schedule them as a recurring "desired-state"
  policy. Re-runs do **not** rotate the password (see §5).

## 3. Per-OS rollout steps

The agent CLI contract (verified against this fork) is identical across OSes:
**install → start service → set password → set options → read id → register.**
All privileged calls go over IPC to the running root/SYSTEM service, so the
service must be up first.

### 3.1 Windows — `deploy/provision-windows.ps1`

```powershell
$env:NEXTSESSION_API_TOKEN = "<admin-service-account-token>"
powershell.exe -NoProfile -ExecutionPolicy Bypass -File provision-windows.ps1 `
    -InstallerSource "https://files.nxlink.com/nextsession-host.exe" `
    -ApiBaseUrl "https://nextsession.nxlink.com" `
    -SiteTag "site-dallas" -ExtraTags @("rmm","ninja") `
    -ServiceUserId <svc_user_id> -CollectionId <cid>
```

Does, idempotently: silent-install as a service (skipped if present) → wait for
Running → set `approve-mode=password` + `verification-method=use-permanent-password`
→ generate a unique 24-char password and set it **only on fresh install** → read
device ID → register in the shared AB (or write the fallback CSV/stdout record).
Exit 0 = success; non-zero = RMM should retry.

### 3.2 Linux — `deploy/provision-linux.sh`

```bash
export NS_API_TOKEN="<admin-service-account-token>"
export NS_AB_USER_ID=<svc_user_id> NS_AB_COLLECTION_ID=<cid>
export NS_API_BASE="https://nextsession.nxlink.com"
export NS_TAGS="site-dallas,rmm,linux"
sudo -E ./provision-linux.sh /opt/rmm/nextsession-*.deb
# or: NS_DEB_URL=https://files.nxlink.com/nextsession.deb sudo -E ./provision-linux.sh
```

Does, idempotently: install the `.deb` if absent (no auto-upgrade, by design) →
enable + start `nextsession.service` (falls back to `--install-service`) → wait
active → generate/set a unique password **once** (root-only marker at
`/var/lib/nextsession-provision/password-set`) → set options → read ID →
register in the shared AB or fall back to CSV (`/var/log/nextsession-provision.csv`)
+ stdout JSON.

### 3.3 macOS — `deploy/provision-macos.sh` + `deploy/macos-mdm-pppc.md`

> **HARD PREREQUISITE — macOS TCC/PPPC.** macOS TCC blocks screen capture and
> input control behind user-granted permissions. **No script, installer,
> `tccutil`, or root process can grant them** (the TCC DB is SIP-protected;
> `tccutil` can only *revoke*). The **only** unattended grant path is an
> **MDM-delivered PPPC configuration profile** on a **supervised / MDM-enrolled**
> Mac. **The MDM MUST push the PPPC profile before or with the agent, or the
> agent is unusable unattended.** Full payload + identity binding in
> `deploy/macos-mdm-pppc.md`.
>
> - **Accessibility** (control) and **Input Monitoring** (`ListenEvent`): PPPC
>   grants these silently.
> - **Screen Recording** (`ScreenCapture`): Apple treats this specially. On
>   older macOS / non-supervised devices it may **still require one human click**
>   even with PPPC. Validate on your exact OS + supervision state on a wiped Mac
>   before promising "zero touch."
> - Bind every PPPC entry to bundle id `com.nxlink.nextsession` + the app's
>   **Designated Requirement** from `codesign --display -r - /Applications/NextSession.app`.

Deployment order per Mac:

1. **MDM:** push the PPPC profile (`deploy/macos-mdm-pppc.md` §3, with the real DR).
2. **MDM:** push the signed+notarized `.pkg` (postinstall installs the
   LaunchDaemon `_service.plist` and LaunchAgent `_server.plist` —
   `LimitLoadToSessionType=[LoginWindow,Aqua]` is what enables login-window
   capture).
3. **RMM (root):** run the provisioner:

```bash
export NS_API_BASE="https://nextsession.nxlink.com"
export NS_ADMIN_API_TOKEN="<admin-service-account-token>"
export NS_AB_USER_ID=<svc_user_id> NS_AB_COLLECTION_ID=<cid>
export NS_AB_TAGS="site-dallas,rmm,macos"
sudo -E ./provision-macos.sh --pkg https://files.nxlink.com/NextSession.pkg
```

Does, idempotently: install the pkg → ensure LaunchDaemon/Agent loaded → set a
unique password **once** (root-only marker at
`/var/log/nextsession-provision/password-set`) → set `approve-mode=password` →
read ID → register in the shared AB or write the local record JSON + CSV → print
the TCC/PPPC reminder banner.

## 4. Pilot → staged rollout (do NOT push to 1000 at once)

1. **Single-device E2E test (the mandatory §1 gate).** Provision **one** box per
   OS. As a tech, open the shared AB and confirm **one-click connect with the
   saved password works**. If it does not, switch to a §1 fallback before going
   further. This proves the verbatim-password path end-to-end.
2. **Pilot ring (10–25 devices), mixed OS + sites.** Confirm: idempotent re-runs
   don't rotate passwords or duplicate AB entries; the RMM collects the
   fallback record; macOS shows a live (non-black) screen at the login window
   (validates PPPC + Screen Recording).
3. **Staged ring (~100, ~5–10%).** Watch API-server load and the
   `address_book` table growth; confirm 401/403 handling if the token expires.
4. **Fleet (remaining ~900), batched.** Roll in waves (e.g. by site), not all at
   once — the AB `create` calls all hit one API server, and macOS reinstalls the
   pkg per check-in (service restart). Pace the RMM policy.
5. **Reconcile.** Diff collected records against the `address_book` table;
   re-run failures (idempotent).

Rollback: stop the RMM policy. Already-provisioned devices keep working; remove
AB entries via the admin delete endpoint if you need to revoke discovery.

## 5. Security notes

- **Where per-device passwords live:**
  1. The agent's own config — **hashed/encrypted by the agent** (`config.rs`
     `set_permanent_password`, with salt). Not recoverable as plaintext from the
     agent.
  2. The shared AB entry's `password` field — **plaintext to the server**
     (no server crypto). Anyone with read access to the shared collection (every
     tech in the shared group) or DB read can see it. This is the deliberate
     trade for one-click; scope the tech group accordingly.
  3. A **local fallback record** (CSV/JSON + stdout) so the RMM can reconcile:
     - Windows: `%ProgramData%\NextSession\provisioning\` (SYSTEM-only path;
       protect via NTFS ACLs, have the RMM delete after collection).
     - Linux: `/var/log/nextsession-provision.csv` (chmod 600) + marker in
       `/var/lib/nextsession-provision/` (chmod 600, root-only).
     - macOS: `/var/log/nextsession-provision/` (dir chmod 700, files chmod 600).
  > The plaintext password sits in the fallback files and in the AB. Treat all
  > three as fleet secrets. Have the RMM **delete the local fallback files after
  > collection.**
- **Token handling:** the admin token is a **fleet-wide secret** (it can write
  any shared AB). Inject it only via the RMM secret store as an env var; never in
  the policy body or git. The scripts read it from the environment and never log
  it. Rotate it on a schedule; disable it when continuous re-registration is not
  needed.
- **Rotation:** by design the scripts **do not rotate** the per-device password
  on re-run — a marker file (Linux/macOS) / fresh-install gate (Windows) prevents
  it. Rotating in-place would desync the device from its AB entry (the AB
  `create` endpoint rejects the duplicate, so the new password would never reach
  the AB) and break one-click. **Rotation must be a separate, deliberate policy**
  that (a) sets the new password, (b) **updates** the AB entry
  (`PUT /api/ab/peer/update/:guid` or the admin update endpoint), and (c)
  re-collects the record — in one transaction per device.
- **Least privilege for techs:** access is gated by the AB share (`Rule`).
  Off-boarding a tech = remove them from the shared group / drop the rule; the
  per-device secrets stay sealed. No fleet-wide rotation required.

## 6. Known gaps / TODO

Issues found while reviewing the scripts. Fixed-in-place items are marked
**[fixed]**; the rest need a decision or live verification before fleet rollout.

- **[fixed] macOS password rotated on every run.** `provision-macos.sh` had no
  marker gate (unlike Linux's `PW_MARKER` and Windows' fresh-install gate), so
  each RMM check-in generated a **new** password. Because the AB `create`
  endpoint rejects the duplicate (re-runs treat it as "idempotent OK"), the new
  password would never reach the address book → device and AB entry desync →
  one-click connect breaks. Added a root-only `password-set` marker so the
  password is generated/set **once** and reused on re-runs, matching Linux.
- **AB success/duplicate response shape is guessed, not verified.** All three
  scripts infer success/duplicate from HTTP code + substring matching on the
  body (`exist`/`duplicate`/`already`). The exact success and
  duplicate-rejection shape of `POST /api/admin/address_book/create` is **not
  confirmed against the live server** (findings note it rejects duplicates per
  `(user_id,id,collection_id)` but not the response code/body). **TODO:** capture
  one real success and one real duplicate response in the pilot and tighten the
  parsing. Risk: a re-run that silently fails could be logged as "idempotent OK."
- **Verbatim-password client behavior unconfirmed (the §1 caveat).** The server
  stores `password` verbatim; whether a plaintext value yields working one-click
  connect depends on the **client** version and cannot be proven from the repo.
  **TODO:** the §4 step-1 single-device test is the gate. If it fails, use a §1
  fallback.
- **macOS reinstalls the pkg on every run.** `installer -pkg` runs
  unconditionally each check-in (documented as intentional to pick up newer
  pkgs), which **restarts the service on every RMM run** across the fleet. Linux
  deliberately skips reinstall when the package is present. **TODO:** if churn is
  a concern, gate the macOS install on a version check or a "pkg already
  installed at version X" guard, and pace the RMM policy.
- **Linux: no auto-upgrade.** `install_deb` skips if the package is present (by
  design, to avoid surprise fleet-wide restarts). **TODO:** upgrades need a
  separate, paced policy with a version comparison.
- **macOS LaunchDaemon label may differ from the plist basename.** The
  `launchctl print system/<bundle>_service` label is assumed to equal the plist
  basename; the script already flags this as `TODO(verify)` and degrades to a
  warning. **TODO:** confirm the real label on the target macOS version and
  tighten `ab_wait_for_service`.
- **PPPC profile placeholders.** `deploy/macos-mdm-pppc.md` ships with
  `REPLACE_TEAM_ID` / `REPLACE-WITH-UUID` and a placeholder Designated
  Requirement. **TODO:** paste the real DR from
  `codesign --display -r - /Applications/NextSession.app` and confirm whether the
  root daemon binary is inside the bundle (else add a `path`-based PPPC entry).
- **Screen Recording silent-grant is OS/supervision dependent.** May still need
  one human click on older/non-supervised macOS. **TODO:** validate on a wiped
  Mac at the fleet's exact OS + DEP state (`macos-mdm-pppc.md` §4).
- **Default `ServiceUserId`/`CollectionId` are `0`.** Windows defaults both to
  `0`; `0` collection means the service user's *personal* AB, and
  `ServiceUserId<=0` makes the script fall back instead of registering. **TODO:**
  always pass the real `<svc_user_id>` and `<cid>` from §2.1 — do not rely on
  defaults.
- **`username` is hardcoded `"admin"`** in the Windows/macOS AB payloads (Linux
  sends empty). Cosmetic (it is the entry's display username, not auth).
  **TODO:** set it to something meaningful or leave empty consistently.
- **No hardcoded secrets found.** All three scripts read the admin token from the
  environment and never log it; servers/key are correctly left to `custom.txt`.
  Generated passwords use a CSPRNG and are not echoed to logs. Good.
